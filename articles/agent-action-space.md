# Agent Action Space

——我们耗费心血让模型拥有了跨越山海的算力，却又在它脚下画了一个必须按步踩踏的格子。

Anthropic的Claude Code的核心开发Thariq Shihipar发布一篇文章“Lessons from Building Claude Code: Seeing like an Agent”，摘要里直接提到：构建 agent harness 最难的部分之一，是设计它的 action space；在做 AskUserQuestion 工具时，他们的目标是提升 Claude 的主动澄清能力（elicitation）。

我分三层给你拆。

## 1）它表面在讲工具设计，实质在讲“约束如何塑造智能”

从外部引用和二次总结里，这篇内容反复出现几个关键词：

`action space beats action power`：不是工具越强越好，而是行动空间越清晰越好。一个边界清楚的 3 工具 agent，可能比一个 12 工具但边界模糊的 agent 更可靠。

工具不是中性的能力集合，它会反过来决定模型如何拆解问题、如何形成策略。工具越多，选择熵越高，模型越容易走偏。

AskUserQuestion 的出发点不是“加个交互 UI”，而是解决 agent 在不确定时该停下来问人，而不是继续脑补的问题。

这点很关键。很多人理解 agent 时，默认思路是：

模型 + 更多工具 = 更强 agent

但这条内容强调的其实是相反方向：

模型 + 设计得当的可选动作集合 = 更稳 agent

这不是字眼游戏，而是工程现实。Anthropic 自己公开写过，做 agent 时，他们在某些场景里花在工具优化上的时间比花在总 prompt 上还多；比如他们发现相对路径会让模型频繁犯错，于是直接把工具改成“永远要求绝对路径”，错误率就明显下降。

所以这条推文最值得抓住的，不是“Claude Code 会提问”，而是：

模型并不会自动把你提供的能力集变成可用策略。能力需要被“编排成模型看得懂的环境”。

## 2）它真正有价值的地方：承认“今天的最佳设计，明天可能变成枷锁”

这篇内容之所以在圈里传播，不是因为结论新，而是因为它公开承认了一件很多团队不愿承认的事：

工具设计和模型能力是共演化的。

外部评论里对这篇的提炼很一致：
“what works for one model might constrain the next”——对旧模型有效的工具设计，到了新模型时代，可能反而开始限制它。

这背后有个很容易被忽略的悖论：

早期模型弱

你会倾向于：

- 加更多显式步骤
- 做更细的专用工具
- 用更强的护栏
- 通过结构化提问来兜底

后期模型强

这些东西可能变成：

- 冗余决策节点
- 不必要的强制流程
- 过拟合某一代模型行为的“旧脚手架”
- 压制模型更自然策略的束缚

所以这条内容真正高级的地方，不是“教你怎么设计工具”，而是提醒你：

agent 产品不是一次性设计，而是持续校准。

这和 Anthropic 其他公开工程文章是对得上的。比如他们在多智能体研究系统里明确写过：要“像 agent 一样思考”，通过逐步观察 agent 的真实行为来暴露失败模式，例如：

- 结果已经足够还在继续搜
- 搜索词过于冗长
- 选错工具

他们是靠看 agent 一步步怎么做来发现问题，而不是只看最终答得对不对。

这说明一个很现实的工程原则：

Agent 的问题，很多不是离线设计能解决的，而是要通过行为观测去修正。

## 3）“好工具”的设计阵型：把模式压缩成可复用的架构判断

为了形成高效的设计模式传递，我们可以将这个工具的架构核心压缩为以下对比矩阵：

| 设计维度 | 灾难性的设计方案 | 顶级设计（Claude Code 范式） | 架构认知价值 |
| --- | --- | --- | --- |
| 触发时机 | 作为常规回答的后缀（挂载式提问） | 主动熔断：遇到技术路径歧义，立刻阻断执行 | 用“停顿的留白”彻底消除代码重构的幻觉成本。 |
| 交互形态 | 开放式问答：“你想要什么？” | 收敛式阵型：强迫模型输出结构化的 Options（单选/多选） | 迎合人类强大的视觉并行处理能力，通过选项降低大脑的认知带宽消耗。 |
| 作用域 | 允许子 Agent（Subagents）随意调用 | 仅限主线程：`Task()` 生成的无状态子 Agent 被严格剥夺该权限 | 保持工作流的状态机清晰，防止子进程在后台陷入无人响应的“幽灵死锁”。 |

这张表真正强调的是：好工具不是“能力更全”，而是“动作边界更干净”。

如果把它翻译成 agent 设计语言，本质上就是三件事：

- 什么时候必须停下来，把不确定性交还给人；
- 什么时候必须收缩选择空间，避免模型把开放问题越问越散；
- 什么时候必须限制权限边界，防止多线程或多 agent 结构把流程状态搞乱。

这也是为什么很多团队明明工具更多、流程更复杂，结果却没有更稳定。问题不在于他们没有能力，而在于他们没有把能力组织成一个低歧义、低漂移、低死锁风险的行动空间。

## 4）我对这条内容的判断：

### 我认可的部分

这条内容很重要，因为它把注意力从“模型崇拜”拉回到了“环境设计”。

Anthropic 官方自己的 agent 经验也支持这一点：

- coding agent 之所以有效，一个重要原因是有可验证反馈回路，比如测试结果；
- tools 的定义和格式，本身就要像 prompt engineering 一样认真做；
- multi-agent 系统能提效，核心也不只是“多线程”，而是把任务拆到不同上下文窗口里，从而扩展有效 token 与搜索容量。

所以这条贴文背后的原则可以浓缩成一句：

Agent 不是模型自己长出来的，而是“模型 + 工具接口 + 反馈机制 + 人机边界”共同形成的系统。

### 但我会质疑的部分

这类内容也很容易被社区过度提炼成口号，比如：

- “less is more”
- “不要多工具”
- “工具设计比模型更重要”

这些都不完整。

因为真正成立的前提是：
你面对的任务类型、模型代际、验证闭环、上下文预算，都得匹配。

Anthropic 自己在 Research 系统里又展示了另一面：
当任务需要并行探索、信息超出单上下文窗口、工具链复杂时，多 agent 反而显著更强；他们内部评估里，多 agent Research 系统相对单 agent Opus 4 有 90.2% 的提升，但代价是 token 消耗大幅上升，multi-agent 通常约为普通 chat 的 15 倍 token。

这说明什么？

说明“少工具更好”不是普适规律。
真正的规律是：

工具复杂度必须和任务结构、模型能力、成本约束一起配平。

## 5）如果把它翻译成更关心的产品/平台语言，结论是什么

这条内容最值得吸收的是下面 5 点：

### ① 不要先问“还能接什么工具”，先问“模型在什么分叉点会犹豫”

很多团队做 agent 的第一反应是扩能力边界。
但实际更该先看：

- 模型在哪些节点容易误判？
- 哪些动作选择会让它分心？
- 哪些信息缺口必须回问用户？
- 哪些操作需要强约束而不是自由生成？

AskUserQuestion 本质就是把“澄清”提升为一等公民动作。

### ② 工具描述本身就是“行为引导层”

Anthropic 明确写过：tool definitions 应该像 prompt 一样精心设计；参数名、边界、示例、格式，都直接影响模型使用质量。

也就是说：

- MCP 不是接上就完了
- Skill 不是写个说明文档就完了
- Tool schema 不是 API 文档复制过来就完了

### ③ 你设计的不是“功能”，而是“可被模型理解的 affordance”

同样一件事，让模型写 diff、写整文件、输出 JSON、输出 markdown，对软件来说等价，对模型来说却不等价。Anthropic 直接说了：有些格式对 LLM 更难。

所以产品经理/平台工程师要开始接受一个事实：

机器可执行，不等于模型可使用。

### ④ 要把“观察 agent 怎么失败”做成正式方法，而不是临时 debug

Anthropic 在多 agent 文里说得很明确：他们会用和生产一致的 prompt/tool 去模拟，然后逐步观察 agent 行为。

这意味着未来 agent 平台真正重要的能力可能不是“接更多模型”，而是：

- 行为轨迹观测
- 工具调用分析
- 失败路径聚类
- action entropy 监控
- prompt/tool 版本对比实验

### ⑤ 你的平台如果不支持“随模型演进而重构工具层”，后面会很痛

这是我最看重的一点。
这条内容本质在提醒：

产品里很多“好用的设计”，其实只是暂时适配当前模型。

所以平台层要支持：

- 工具版本化
- 针对不同模型的 action-space 配置
- 低成本 A/B 测试
- 可回滚的 tool schema
- 将 Skill / MCP / 原生工具做统一策略编排

否则一旦模型升级，你原来的“最佳实践”会直接变成负资产。

## 6）我给这条内容的最终评价

它不是在讲一个功能点

而是在讲 agent engineering 的第一性原理：

要从“用户看到什么能力”切到“模型看到什么世界”。

它最有价值的洞察

不是“Claude Code 很强”，而是：

工具会塑造模型的思维路径；而当模型变强时，昨天的辅助结构会变成今天的阻碍。

它对做平台的人最有启发

不是去抄一个 AskUserQuestion，而是反问：

- 我现在给 agent 的 action space，是不是过宽了？
- 哪些自由度其实应该前移为结构化选择？
- 哪些工具是为了补旧模型的短板，今天已经该删了？
- 哪些能力应该做成 MCP，哪些应该保留成更强约束的原生工具？
- 我们有没有观测 agent 行为而不是只看结果？
