# 从 todo 到 task

Todo会失忆，先别把锅全甩给模型“笨”。

## 背景：这不是简单改名，而是工作组织模型升级

这条 “We’re turning Todos into Tasks in Claude Code” 讲的不是简单改名，而是一次底层工作组织模型的升级。

它的核心意思可以概括成：

Claude Code 把原来的 Todos 升级成 Tasks。Tasks 是一种新的基础原语，用来帮助 Claude Code 跟踪并完成更复杂的项目，还能让它们在多个 session 或 subagents 之间协作。

也就是说，原来的 Todo 更像是：

- 当前会话里的待办列表
- 主要作用是“别跑偏，记住接下来要做什么”

而新的 Task 更像是：

- 可持久化的任务对象
- 带状态、依赖、阻塞关系
- 可以跨会话、跨子 agent 共享
- 更像一个轻量级 agent work graph

外部转述里还引用了他们后来对设计意图的补充：Todos 更像是让模型保持在轨道上；Tasks 更偏向于帮助 agents 彼此沟通。

### 和以前的 Todo 相比，变化在哪

#### 1）从“会话内提醒”变成“持久化任务系统”

Tasks 会存到本地文件系统里，路径是 `~/.claude/tasks`。这意味着它不是上下文里一段容易被压缩掉的文本，而是会话外部状态。

这点很关键。

因为 Todo 最大的问题不是“不能列计划”，而是：

- 上下文压缩后容易丢
- 关闭 session 后断掉
- 多轮长任务里，模型会忘掉之前拆过什么
- 多 agent 并行时很难同步

Tasks 就是在修这个。

#### 2）从“线性清单”变成“带依赖关系的任务图”

Tasks 不只是一个列表，它支持依赖和 blocker 元数据。外部介绍里明确提到，task 可以有 `blocked by` 关系。

这意味着 Claude Code 开始从：

先做 A，再做 B，再做 C

升级到：

- B 依赖 A
- C 被 B 阻塞
- 哪些任务可以并行，哪些不行

这不是 UI 小升级，而是把“计划”从文本结构变成了可执行协调结构。

#### 3）从“单 agent 自我管理”变成“多 agent 协作介质”

他们强调 Tasks 支持多个 session 或 subagents 协作，而且更新会广播给同一个 task list 上的其他会话。还可以通过环境变量 `CLAUDE_CODE_TASK_LIST_ID=name claude` 共享同一个任务列表。

这说明 Tasks 的真正目标并不是让一个 Claude 更会列清单，而是让：

- 主 agent 拆任务
- 子 agent 接任务
- 多个 session 共享进度
- 任务状态在外部一致

也就是说，它把原来隐含在 prompt 里的协调逻辑，开始外显成了系统原语。

### 为什么他们要做这个

从公开内容看，原因很明确：

#### 第一，复杂项目已经超出“Todo 列表”能力

Tasks 被定义为适合 more complicated projects。比如多文件重构、迁移、长周期 feature 开发。

Todo 适合短任务；Task 适合中长程执行。

#### 第二，Claude Code 明显在往“长时运行 agent”发展

Tasks 跨会话持久化，意味着他们已经不再把 Claude Code 只当成一次性交互 CLI，而是在往下面这个方向推进：

- session 可中断再继续
- context 可以压缩，但任务状态不丢
- agent 可以协作
- 工作可以拆成长期项目轨迹

这本质上是在补 agent 产品最难的一块：外部状态管理。

#### 第三，他们在把“计划”从 prompt 技巧升级成产品能力

以前很多 agent 的“计划能力”其实是假象：模型在回复里写一段 plan，看起来像有规划，但实际上没有真正的状态机。

Tasks 的意义就是：

- 计划不再只是语言
- 计划变成有状态、有生命周期的对象
- 系统能围绕它构建新能力

这也是为什么外部解读会说：Tasks 是一个 new primitive。

### 它隐含了什么产品方向

我觉得这条最值得分析的，不是功能本身，而是它暴露出的 Claude Code 产品路线。

#### 1）Claude Code 在把“上下文”外部化

很多人以为 coding agent 的核心是更大 context window。但 Tasks 说明他们在走另一条路：

不是一味把所有历史塞进上下文，而是把稳定状态抽到上下文外。

这比纯靠长上下文更工程化，也更可扩展。

#### 2）Claude Code 在把“协作协议”产品化

如果多个 subagents 都围绕同一个 task list 工作，那 task 实际上已经是：

- 协作接口
- 状态同步层
- 分工协议

这和简单 Todo 完全不是一个等级。

#### 3）它在为更复杂的 agent 编排铺路

一旦 task 成为正式原语，后面自然就能长出更多东西，比如：

- task priority
- task owner
- task retry
- task artifact linkage
- task-level approval
- task-specific context injection

虽然这条帖子没说这么多，但从 primitive 设计看，这个方向几乎是顺着长出来的推论。这个属于我的推断，不是原帖明说。原帖明确支持的只有“跨 session / subagent 协作、依赖关系、持久化”。

### 它对使用者意味着什么

如果你是普通用户，这次更新的实际意义大概是：

#### 更适合长任务

比如：

- 大型重构
- 跨多个目录改造
- 分阶段迁移
- 连续几天推进的 feature

以前 Todo 很容易中途“失忆”，现在 Task 更适合连续推进。

#### 更适合多窗口 / 多 agent 并行

如果你同时开多个 Claude Code session，或者让 subagent 分头做事，Tasks 提供了共享任务状态的基础。

#### 更不怕上下文压缩

因为任务状态不再只靠对话历史保留，而是落盘。外部总结明确把这点当成核心收益之一。

### 我对这条更新的判断

我觉得这次升级非常重要，原因不是“Todo 变高级了”，而是它说明 Claude Code 正在从：

会聊天的 coding assistant

往

有外部状态、有协作结构、有长期执行能力的 agent runtime

转。

也就是说，真正的变化不是 UI 词汇从 Todo 变成 Task，而是 Claude Code 开始承认一件现实：

复杂软件开发不是一次推理问题，而是持续协调问题。

Todo 更偏“记住我要做什么”。
Task 更偏“在一个持续演进的工程系统里，管理谁在什么时候完成什么”。

### 一句话总结

这条博客的本质是：

Claude Code 不再只是让模型写一个待办清单，而是在给 agent 引入一个可持久化、可协作、可表达依赖关系的任务层。

这一步很像是把“prompt 里的计划”升级成“系统里的状态机雏形”。

Todo 会失忆，本质上不是记忆功能坏了，而是 Todo 常常只是“上下文里的任务描述”，不是“系统外部的稳定状态”。

一旦对话被压缩、切换窗口、分派给子 agent，或者长任务跨越多个上下文窗口，原来那份 Todo 很容易被摘要化、弱化，甚至丢掉细节。Anthropic 自己在长时运行 agent 的文章里就明确说了：Agent SDK 依赖 compaction 来避免耗尽上下文，但 compaction 本身并不充分；模型跨多个上下文窗口连续工作时，单靠这套机制仍然不足以支撑复杂长期任务。Claude Code 的 changelog 里也单独提过，他们曾专门修过 “compaction 期间的 todo list handling”，这等于侧面承认了 Todo 和上下文压缩之间本来就有张力。

再说得更尖锐一点：

Todo 天然更像“给当前这次思考看的便签”，而不是“给整个系统长期保存的任务对象”。

Agent SDK 的 Todo 文档把 Todo 定义成一种帮助组织复杂流程、向用户展示进度的结构，生命周期也非常轻：`pending -> in_progress -> completed -> 全部完成后移除`。这个设计本来就偏“过程可视化”和“短期组织”，不是为长期项目记忆、跨 session 协作、复杂依赖图而生。也就是说，Todo 的第一性目标不是“永不遗忘”，而是“让当前 agent 保持在轨道上”。

所以“失忆”其实有三层来源。

## 第一层：上下文不是数据库

LLM 的工作记忆是 prompt window，不是持久化存储。你把 Todo 写在上下文里，本质上还是“下一轮推理可见的文本”，不是 durable state。只要进入压缩，系统就会优先保留它认为最重要的信息，而不是逐字逐句保留原始 Todo。

Anthropic 关于 context engineering 的文章本质就在强调：复杂 agent 往往要做关注点分离，不能把所有细节都留在同一个上下文里；多 agent 的价值之一，就是把局部细节隔离在各自窗口内，让主 agent 只处理综合结果。这个原则反过来也说明，纯靠一个共享上下文里的 Todo 去承担全部状态，迟早会出问题。

## 第二层：Todo 是“线性提醒”，不是“结构化任务图”

它适合表达“接下来做 A/B/C”，不适合表达“B 依赖 A、C 被 D 阻塞、E 由另一个 agent 负责”。

一旦任务复杂到需要依赖、并行、阻塞、回滚、跨会话继续，Todo 这种线性清单就会开始塌。塌了以后，看起来像“Claude 忘了”，其实更准确地说，是原始表示法已经装不下任务真实结构了。

Anthropic 对 subagent 的官方描述也说明了这点：子 agent 运行在独立上下文窗口里，拥有自己的 prompt、工具和权限。既然上下文本来就是分裂的，一个只存在于当前主会话语境中的 Todo，天然不够做跨 agent 协调。

## 第三层：Todo 的设计哲学本来就不是“全能记忆层”

很多人误以为 Todo 是 memory system，其实更接近 attention control system。它的作用首先是让 agent 当前阶段别跑偏、让用户看到进度、帮助模型把大任务拆成可执行小步，而不是替代文件系统、数据库、任务队列、事件日志这些外部状态层。

Anthropic 在 “Building Effective Agents” 里一直强调的是：高效 agent 往往来自简单、可组合的模式，而不是一开始就堆复杂框架；在 “Writing effective tools” 里又强调，工具设计会直接决定 agent 的表现。把 Todo 保持成轻量原语，本身就符合这种哲学：先用最小结构维持秩序，而不是一上来就把它设计成重型项目管理系统。

## 这里面的设计哲学，核心有 5 个

### 1）“记忆”要分层，不能都塞给模型

一个成熟 agent 系统通常至少分三层：

- 工作记忆：当前上下文里正在推理的内容
- 任务状态：当前有哪些事待做、做到哪了
- 事实记忆 / 产物记忆：代码、文件、日志、数据库、artifact

Todo 只适合第二层里最轻的一部分，甚至更像第二层的 UI 投影。

一旦你希望它承担第一层和第三层，它就会表现成“失忆”。这不是 bug，而是越权使用。Anthropic 对 bash tool 的描述都比 Todo 更强调 “persistent session that maintains state”；这恰恰说明真正的状态持久化，应该落在外部执行环境，而不是只落在文本列表里。

### 2）“压缩不可避免”，所以必须接受信息会被重写

长时 agent 不可能无限扩上下文，只能压缩。

而压缩的本质就是：牺牲原始细节，保留被系统判断为重要的抽象。这意味着任何只存在于上下文文本中的 Todo，都注定可能被重写。Anthropic 明说 compaction 是长任务的必要能力，但也明说它不够。这个“不够”，本质就是：压缩能维持连续性，不能保证精确保真。

### 3）“表示法决定能力上限”

如果你的任务表示法只是 checklist，你得到的就主要是 checklist 级智能。

你很难从一份线性 Todo 里自然长出：

- 依赖管理
- 资源协调
- ownership
- 并行执行
- 阻塞恢复
- 跨 session 续跑

这就是为什么后来从 Todo 走向 Task 是合理演化：不是改名，而是换表示法。旧表示法的“失忆”，其实是在提醒你它的抽象层级太低了。

### 4）“上下文是昂贵资源”，所以能外部化的就外部化

Anthropic 在多 agent / context engineering 文章里一直强调，要把细节隔离、把主上下文留给更高价值的综合判断。

这背后的哲学很明确：

不要让模型用最贵的资源，去保存最机械的状态。

Todo 如果长期占据上下文，本质上是在浪费高价值 token。

更合理的做法是：

- 任务状态外部化
- 中间产物文件化
- 长链路信息日志化
- 当前窗口只加载“此刻需要的最小充分上下文”

### 5）“好 agent 不是全记住，而是知道该把什么记在哪”

这是最重要的一点。

很多人把 agent 的高级感理解成“什么都记得”。其实真正成熟的系统更像一个组织：

- 短期思考放脑子里
- 计划写在任务系统里
- 代码写在仓库里
- 证据留在日志里
- 决策依据沉淀成文档
- 子团队各自带局部上下文

也就是说，智能不是靠一个超大脑袋，而是靠正确的状态分层与边界设计。

## 那么为什么 Todo 仍然值得存在

因为它虽然会“失忆”，但它在自己的定位上仍然很有价值：

- 它让模型显式拆步，减少无计划乱跑
- 它让用户可见当前进展
- 它给 agent 一个低成本的自我约束结构
- 它比完全自由生成更容易被监控和 debug

换句话说，Todo 的价值从来不是“永恒记忆”，而是短程控制。

## 一句更本质的话

Todo 会失忆，不是因为它设计失败，而是因为它被设计成“让 agent 保持方向”的轻量机制，而不是“替代外部状态系统”的重型机制。

所以它背后的哲学不是“模型要像人一样全记住”，而是：

把最该持久化的东西交给系统，把最该灵活推理的东西交给模型。

这才是 agent 设计里更成熟的路线。
